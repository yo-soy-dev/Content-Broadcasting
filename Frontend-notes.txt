FRONTEND NOTES — Pulseboard Content Broadcasting System
==========================================================

1. PROJECT STRUCTURE
--------------------
Feature-based folder structure with strict separation of concerns:

src/
├── app/          → Next.js App Router pages grouped by role
│                   auth/login, auth/register
│                   teacher/dashboard, teacher/upload, teacher/my-content
│                   principal/dashboard, principal/approvals, principal/all-content
│                   live/[screenId]  ← public, no auth required
├── components/   → Divided by role: teacher/, principal/, shared/, layout/, ui/
├── services/     → ALL API calls go through here only (enforced by architecture)
├── hooks/        → Custom hooks wrap services, expose loading/error/data state
├── context/      → AuthContext — global user state and JWT token
├── types/        → TypeScript interfaces for Content and User
├── constants/    → Static data — SCREEN_OPTIONS, SUBJECTS (no API call needed)
├── utils/        → Pure helper functions — dateHelpers, validators, statusHelpers
└── lib/          → axios instance config + React Query client

Why this structure:
- Single responsibility per folder — easy to find and change code
- Components stay thin — no API logic, no token logic
- Evaluator can trace any data flow: page → hook → service → axios → API
- Scalable — add new features without modifying existing ones


2. AUTHENTICATION FLOW
----------------------
Step-by-step:

1. User submits LoginForm → useAuth hook → authService.login() called
2. API returns { token, user } → token saved to localStorage
3. user object saved to AuthContext state
4. axios interceptor auto-attaches token to every subsequent request
5. On 401 response → interceptor clears token → redirects to /auth/login
6. On page refresh → AuthContext mounts → calls authService.getMe()
   → rehydrates user from token (session persists across refresh)
7. AuthGuard wraps all protected pages → checks isAuthenticated + role

Token storage: localStorage (acceptable for assignment scope)
Session persistence: getMe() on every app mount


3. ROLE-BASED ROUTING
---------------------
Flow after login:
- loggedInUser.role === 'teacher'   → router.push('/teacher/dashboard')
- loggedInUser.role === 'principal' → router.push('/principal/dashboard')

Protection:
- src/app/teacher/layout.tsx   → AuthGuard allowedRoles={['teacher']}
- src/app/principal/layout.tsx → AuthGuard allowedRoles={['principal']}

AuthGuard behavior:
- Not authenticated         → redirect to /auth/login
- Wrong role (e.g. teacher visiting /principal) → redirect to own dashboard
- Loading state             → show branded spinner (not blank screen)

Public routes (no AuthGuard):
- /auth/login, /auth/register
- /live/[screenId]          → students visit directly, zero authentication


4. API INTEGRATION APPROACH
---------------------------
All API calls are isolated in services/ folder:

  auth.service.ts     → login(), register(), getMe(), logout()
  content.service.ts  → upload(), getMyContent(), getStats(), deleteContent()
  approval.service.ts → getPending(), getAllContent(), approve(), reject()

Rules enforced:
  ❌ No axios calls inside components
  ❌ No axios calls inside hooks directly (hooks call services)
  ✅ Components call hooks only
  ✅ Hooks call services only
  ✅ Services call axios only

axios instance (lib/axios.ts):
  - baseURL from NEXT_PUBLIC_API_URL environment variable
  - Request interceptor: attaches Bearer token from localStorage
  - Response interceptor: catches 401 → clears token → redirects to login
  - 10s timeout: prevents UI hanging on slow responses
  - Easily replaceable: change NEXT_PUBLIC_API_URL → entire app uses new backend


5. STATE MANAGEMENT
-------------------
Three clean layers, no Redux needed:

  a) Server State → TanStack Query (React Query)
     - useQuery  → fetch and cache data (content lists, stats)
     - useMutation → create/update/delete with onSuccess cache invalidation
     - Automatic background refetch keeps UI fresh
     - staleTime configured to reduce unnecessary requests

  b) Global Auth State → AuthContext (React Context)
     - Stores: user, token, isLoading, isAuthenticated
     - Single source of truth for auth across all components
     - Rehydrates on mount via getMe()

  c) Local UI State → useState
     - Search input value, active filter status, modal open/close, page number
     - Kept minimal — only per-component transient state

Decision: No Redux, no Zustand.
React Query handles all async server state.
AuthContext handles the only global client state needed.
This is the correct minimal approach for this application size.


6. FORM HANDLING & VALIDATION
------------------------------
Libraries: React Hook Form + Zod

Validation schemas in utils/validators.ts:

  loginSchema:
    - email: required, valid email format
    - password: required, min 6 characters

  uploadSchema:
    - title: required
    - subject: required (from SUBJECTS constant)
    - file: JPG/PNG/GIF only, max 10MB (client-side before API call)
    - startTime: required
    - endTime: required, must be strictly after startTime (Zod .refine())
    - screen: required

File validation happens in handleFile() before form submission.
Invalid files never reach the network.
File preview shown before upload via URL.createObjectURL().
Submit button disabled and shows spinner during upload.


7. SCREEN SELECTION — DESIGN DECISION
---------------------------------------
Screens are hardcoded in constants/formOptions.ts as SCREEN_OPTIONS array.

Reason:
Screens represent fixed physical display monitors in school building rooms.
They do not change dynamically at runtime.
Creating a separate Screen collection, API endpoint, controller, and route
for completely static data is over-engineering.

A constants file is simpler, faster, and correct for this use case.
Also avoids an extra API call on every UploadForm load.

If screens need to be dynamic in future → replace constant with useScreens() hook.
The change would be isolated to UploadForm only.


8. PERFORMANCE
--------------
- React Query caching: identical queries return cached data instantly
- Debounced search (400ms): API called only after user stops typing
- Skeleton loaders: shown during fetch — no blank/jumping screens
- Pagination: All Content page loads 10 items at a time (not all 1000+)
- Fixed image dimensions with object-cover: prevents layout shift
- useMemo/useCallback: used where re-render cost is measurable
- Mobile card layout: avoids rendering large tables on small screens
- React Query staleTime: reduces background refetch frequency


9. EDGE CASES HANDLED
----------------------
Scenario                     → Handling
─────────────────────────────────────────────────────────
Empty content list           → EmptyState component with icon + CTA
API fetch error              → ErrorState with message + retry button
Slow API response            → Skeleton loaders, buttons disabled on submit
Invalid login credentials    → Inline error message below form
Upload failure               → Toast error, form stays filled (not reset)
Invalid file type            → Client validation rejects before API call
File too large (>10MB)       → Client validation rejects before API call
401 Unauthorized response    → Auto-logout + redirect via axios interceptor
Broken/missing API fields    → Optional chaining everywhere (data?.field ?? fallback)
No live content available    → "No content broadcasting right now" empty state
Network error on live page   → Error screen with refresh button
Teacher visits /principal    → AuthGuard redirects to /teacher/dashboard
Principal visits /teacher    → AuthGuard redirects to /principal/dashboard
Unauthenticated page visit   → AuthGuard redirects to /auth/login
Invalid teacherId in URL     → Backend validates ObjectId format, returns []


10. ASSUMPTIONS
---------------
- JWT stored in localStorage (acceptable scope for this assignment)
- File types: JPG, PNG, GIF only — max 10MB (per assignment spec)
- Cloudinary used for file storage — not local disk
- Screen is a String field on Content model — not a separate DB collection
- Subjects list is static — not fetched from API
- Live page polls every 30 seconds via React Query refetchInterval
- Content is "active" when: status=approved AND startTime <= now <= endTime
- Only pending content can be deleted (by teacher)
- Only pending content can be approved or rejected (by principal)
- TypeScript used instead of plain JavaScript — better DX, safer code
- Dark mode implemented via Tailwind dark: classes + next-themes
